GEJAST v665 — Push / service worker / dispatcher action-token runtime completion

Goal:
Complete the next concrete implementation step after v664 drinks runtime: the push/action-token path used by drinks verification notifications.

Implemented behavior:
- Browser registers service worker, requests notification permission, subscribes to push, syncs subscription to Supabase, and touches active presence with player session, platform, install mode, permission, page path and optional geolocation.
- Drinks runtime can queue nearby verification push jobs through queue_nearby_verification_pushes_v3.
- Nearby queueing writes actionable web_push_jobs rows only for active, same-scope, permission-granted nearby users with usable subscription keys and location proximity.
- Dispatcher claims queued jobs, mints verify/reject action tokens when needed, sends web push payloads with both camelCase and snake_case token fields, and marks jobs sent/failed.
- Service worker reads both camelCase and snake_case payload fields, shows verify/reject actions when tokens are present, consumes action tokens on notification click, and opens the correct pending/speed page with result query params.
- SQL consume_web_push_action_v3 applies the action to drink event or speed verification using existing verification owners / v664 contract fallback.

Boundary:
This completes static frontend + SQL + dispatcher implementation for the push action-token path. Live proof still requires deploying the files, running the SQL, and running the dispatcher with service-role/VAPID private key secrets.
