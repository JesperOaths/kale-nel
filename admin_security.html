Security & hosting

Deze pagina is bijgewerkt naar de huidige situatie: statische publieke site op kalenel.nl, admin-RPC’s in Supabase, en de wens om admin/vault/leaderboard echt af te schermen voor live gebruik.

Wat op dit moment nog wél en níet klopt

  • Klopt nog steeds: front-end session gates, noindex,nofollow, en force-HTTPS scripts zijn nuttig als UX-laag.
  • Niet genoeg als echte beveiliging: zolang admin-bestanden rechtstreeks als statische HTML publiek bereikbaar blijven, kan iemand met de URL die pagina nog steeds proberen te openen.
  • DB-checks blijven verplicht: gevoelige Supabase RPC’s moeten admin-sessie server-side blijven controleren, ook ná hosting-level afscherming.

Aanbevolen live plan: Cloudflare vóór de admin-laag

  1. Laat de publieke site op de root staan (kalenel.nl) voor index.html, request.html, activate.html, login.html en eventueel scorer.html.
  2. Maak een aparte admin-ingang, bij voorkeur admin.kalenel.nl of een beschermde /admin/*-route.
  3. Zet Cloudflare voor het domein en bescherm die admin-subdomain of route met Cloudflare Access policies.
  4. Verhuis alleen admin-intent pagina’s achter die Access-laag: admin.html, admin_claims.html, admin_expired.html, admin_reserved_names.html, admin_security.html, vault.html en desgewenst leaderboard.html.
  5. Laat Supabase admin-RPC checks bestaan als tweede slot op de deur.

Alternatieven

  • Netlify Identity: bruikbaar als je die admin-surface naar Netlify verhuist.
  • Vercel middleware: bruikbaar als je admin-routes op Vercel host en route-level auth wilt afdwingen.
  • Reverse proxy auth: goed als je al een eigen proxy of serverlaag voor het admin-pad wilt beheren.

Voor de huidige architectuur is Cloudflare het meest logische eerste pad, omdat het goed past vóór een bestaande statische site met custom domein.

Operationele checklist vóór live testen

  • Zet GitHub Pages HTTPS enforcement aan voor het domein.
  • Verifieer het custom domein bij GitHub Pages.
  • Haal admin-links uit publieke navigatie waar niet nodig.
  • Zet admin/vault/leaderboard achter de gekozen hosting-auth laag.
  • Controleer daarna opnieuw dat admin-RPC’s nog steeds alleen met geldige admin-sessie werken.
