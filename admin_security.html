Beveiliging & hosting

Dit is nu het echte stappenplan voor het afschermen van de admin-hub en alle admin-subpagina’s. Doel: de publieke site publiek houden, maar de admin-laag alleen bereikbaar maken voor jou via een extra toegangspoort vóór GitHub Pages.

Huidige situatie

  • Goed als tweede slot: admin-sessies in Supabase, RPC-validatie, en noindex,nofollow.
  • Nog niet goed genoeg: zolang admin.html en de andere admin-bestanden als gewone statische pagina’s op hetzelfde publieke domein staan, zijn ze nog steeds direct opvraagbaar met de URL.
  • Conclusie: de admin-pagina’s moeten achter een hosting-laag komen die eerst identiteit controleert en pas daarna de HTML doorlaat.

Aanbevolen doelarchitectuur

Publiek houden: index.html, request.html, activate.html, login.html, scorer.html.

Afschermen: admin.html, admin_claims.html, admin_expired.html, admin_reserved_names.html, admin_security.html, vault.html en bij voorkeur ook leaderboard.html.

Voorkeursvorm: verplaats de admin-laag naar admin.kalenel.nl. Dat is schoner en robuuster dan losse losse paden op de root.

Cloudflare-plan in echte stappen

  1. Zet het domein in Cloudflare en zorg dat DNS voor kalenel.nl en het nieuwe admin.kalenel.nl daar beheerd wordt.
  2. Laat de publieke root gewoon wijzen naar de bestaande GitHub Pages-site voor de gewone bezoekersflow.
  3. Maak een aparte admin-hostname zoals admin.kalenel.nl.
  4. Laat die hostname landen op dezelfde statische bundle of op een aparte admin-bundle, maar alleen via een proxied Cloudflare-record.
  5. Maak in Cloudflare Zero Trust een Access-application voor admin.kalenel.nl.
  6. Sta alleen jouw identiteit toe via e-mail, Google-login, One-Time PIN of hardware key. Hou het op één of hooguit twee expliciet toegestane identiteiten.
  7. Zet sessieduur kort voor admin, bijvoorbeeld 8 tot 24 uur, zodat langdurig openstaande browsers minder risico geven.
  8. Verhuis alle admin-links in de hub zodat ze naar de beschermde hostname wijzen, niet meer naar het publieke rootpad.
  9. Laat Supabase admin_check_session en alle admin-RPC’s bestaan als tweede controlelaag. Cloudflare is de buitenpoort; Supabase blijft het binnenslot.
  10. Haal daarna publieke admin-links weg uit de root waar ze niet nodig zijn, of laat ze alleen nog verwijzen naar de beschermde host.

Concreet pad voor jouw site

  1. Publieke site blijft op https://kalenel.nl.
  2. Admin krijgt een nieuwe ingang op https://admin.kalenel.nl.
  3. De bestanden admin*.html, vault.html en leaderboard.html worden vanaf daar geserveerd.
  4. Cloudflare Access laat alleen jouw account door.
  5. Na succesvolle Access-login laad je pas de HTML, daarna pas de Supabase admin-login.

Dat is precies het model: Cloudflare buitenlaag → admin HTML → Supabase admin sessie → admin RPC’s.

Wat je hierna nog moet nalopen

  • GitHub Pages HTTPS enforcement aanzetten.
  • Custom domain bij GitHub verifiëren.
  • Root-HTML geen directe links meer laten geven naar onbeveiligde admin-bestanden.
  • Na de verhuizing testen: directe URL zonder Access, admin hub mét Access, daarna Supabase admin-login, daarna vault en expired queue.

Alternatieven

  • Netlify Identity: goed als je de admin-laag naar Netlify verplaatst.
  • Vercel middleware: goed als je route-level auth in Vercel wilt doen.
  • Reverse proxy: goed als je zelf een kleine server/proxy wilt beheren.

Voor deze site blijft Cloudflare + aparte admin-subdomain de logische route.
